Terminé

🎯 En bref

La POO est un paradigme qui organise le code autour d'objets — des entités autonomes qui regroupent des données (attributs) et des comportements (méthodes). Elle repose sur 4 piliers fondamentaux : encapsulation, héritage, polymorphisme et abstraction.

Analogie

Imaginez une voiture : elle a des attributs (couleur, marque, vitesse) et des comportements (démarrer, freiner, accélérer). En POO, vous créez un "moule" (la classe Voiture) et vous l'utilisez pour fabriquer autant d'instances que nécessaire, chacune avec ses propres valeurs.

Utilisé par

Tous les langages modernes : Java, Python, C#, PHP, JavaScript/TypeScript, Kotlin, Swift, C++.

Les 4 piliers de la POO

PilierDescriptionExemple concret
EncapsulationCacher les données internes, exposer une interface claireAttributs private • getters/setters
HéritageUne classe enfant hérite des propriétés d'une classe parentclass Chien extends Animal
PolymorphismeUn même appel se comporte différemment selon le type de l'objetanimal.parler() → "Woof" ou "Miaou"
AbstractionMasquer la complexité, n'exposer que l'essentielClasses abstraites, interfaces

Exemple en TypeScript

// Classe abstraite → Abstraction
abstract class Animal {
    constructor(protected nom: string) {} // Encapsulation

    abstract parler(): string; // Polymorphisme

    sePresenter(): string {
        return `Je m'appelle ${this.nom} et je dis : ${this.parler()}`;
    }
}

// Héritage
class Chien extends Animal {
    parler(): string { return "Woof !"; }
}

class Chat extends Animal {
    parler(): string { return "Miaou !"; }
}

// Polymorphisme en action
const animaux: Animal[] = [new Chien("Rex"), new Chat("Félix")];
animaux.forEach(a => console.log(a.sePresenter()));
// "Je m'appelle Rex et je dis : Woof !"
// "Je m'appelle Félix et je dis : Miaou !"

Principes SOLID

Les 5 principes fondamentaux pour une bonne conception OOP :

  • S — Single Responsibility : une classe = une seule responsabilité
  • O — Open/Closed : ouvert à l'extension, fermé à la modification
  • L — Liskov Substitution : une sous-classe doit pouvoir remplacer sa classe parente
  • I — Interface Segregation : plusieurs interfaces spécifiques valent mieux qu'une grande interface générale
  • D — Dependency Inversion : dépendre des abstractions, pas des implémentations concrètes

Bonnes pratiques

  • Préférer la composition à l'héritage profond pour plus de flexibilité
  • Respecter l'encapsulation : ne pas exposer l'état interne inutilement (private par défaut)
  • Utiliser les interfaces pour découpler et faciliter les tests (injection de dépendances)
