RabbitMQ

Statut
Terminé
TypeMessage Broker

🎯 En bref : RabbitMQ est un message broker qui permet à des services de communiquer de façon asynchrone en s'envoyant des messages via des files d'attente (queues) — sans que l'émetteur et le destinataire n'aient besoin d'être disponibles en même temps.

Analogie : RabbitMQ est comme La Poste entre tes applications : l'expéditeur (producer) dépose son colis (message) dans une boîte aux lettres (queue), et le destinataire (consumer) le récupère quand il est disponible — sans que les deux doivent se croiser.

Utilisé par : Instagram, Reddit, SoundCloud. Dans toute architecture où l'on veut traiter des tâches en arrière-plan (envoi d'emails, génération de PDF, notifications push…).

Concepts clés

  • Producer : service qui envoie des messages
  • Consumer : service qui reçoit et traite les messages
  • Queue : file d'attente où les messages sont stockés jusqu'à consommation
  • Exchange : routeur qui décide dans quelle(s) queue(s) envoyer un message
  • Binding : règle reliant un exchange à une queue
  • AMQP : protocole de messagerie utilisé par RabbitMQ (port 5672)
  • Ack : accusé de réception — le consumer confirme que le message a bien été traité

Types d'exchange

TypeComportement
DirectRoute vers la queue correspondant exactement à la routing key
FanoutEnvoie à toutes les queues bindées (broadcast)
TopicRoute selon un pattern avec wildcards (*.error, order.#)
HeadersRoute selon les attributs du message header

Exemple de flux

[Service Commande] → Exchange "orders" → Queue "email-confirmation" → [Service Email]
                                       → Queue "stock-update"      → [Service Stock]

Avantages

  • Découplage : les services ne se connaissent pas directement
  • Résilience : si le consumer tombe, les messages restent en queue
  • Scalabilité : plusieurs consumers peuvent dépiler la même queue en parallèle
  • Interface de management : UI web incluse sur :15672 pour monitorer les queues
