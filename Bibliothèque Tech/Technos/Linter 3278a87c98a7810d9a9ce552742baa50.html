Linter

Statut
Terminé
TypeQualité de code

🎯 En bref : Un linter est un outil d'analyse statique du code qui détecte automatiquement les erreurs de syntaxe, les mauvaises pratiques et les violations de style — sans exécuter le programme.

Analogie : Un linter est comme le correcteur automatique de Word, mais pour le code. Il souligne en rouge ce qui ne va pas (erreur de syntaxe), en orange ce qui est douteux (variable non utilisée), et en bleu ce qui ne respecte pas les conventions de l'équipe — en temps réel, pendant que tu écris.

Utilisé par : Toutes les équipes de développement sérieuses, quel que soit le langage. Le linting est une pratique fondamentale de qualité et de cohérence de code.

Ce qu'un linter détecte

  • Erreurs de syntaxe : code qui ne compilera pas ou plantera à l'exécution
  • Variables non déclarées / non utilisées : code mort ou bugs potentiels
  • Mauvaises pratiques : eval(), console.log laissés en prod, comparaisons == au lieu de ===
  • Violations de style : indentation, guillemets, point-virgules, longueur de ligne
  • Problèmes de sécurité : patterns connus pour être risqués

Intégration typique

  • Éditeur (VS Code, IntelliJ) : soulignage en temps réel pendant l'édition
  • Pre-commit hook (Husky + lint-staged) : bloque les commits contenant des erreurs
  • CI/CD : étape lint dans le pipeline avant les tests — bloque les PRs non conformes
# Exemple avec ESLint
npm install --save-dev eslint
eslint . --ext .js,.ts      # Analyser le projet
eslint . --fix              # Corriger automatiquement ce qui peut l'être

Linter vs Formatter

Un linter détecte les problèmes logiques (erreurs, mauvaises pratiques). Un formatter comme Prettier gère l'esthétique (indentation, espaces, retours à la ligne). On les utilise ensemble : le linter pour la qualité, le formatter pour la lisibilité.
