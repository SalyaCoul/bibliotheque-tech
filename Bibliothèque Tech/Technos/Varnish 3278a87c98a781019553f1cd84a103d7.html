Varnish

Statut
Terminé
TypeGestion de cache

🎯 En bref

Varnish Cache est un reverse proxy HTTP et serveur de cache ultra-performant. Placé devant votre serveur web, il mémorise les réponses HTTP et les sert directement depuis la mémoire RAM pour les requêtes suivantes, sans solliciter le serveur d'application ni la base de données.

Analogie

Votre serveur web c'est un chef cuisinier qui prépare chaque plat à la commande. Varnish c'est le serveur en salle qui garde les plats populaires sous une cloche chauffante : quand un client commande le plat du jour (100 fois par heure), il le sert immédiatement sans repasser en cuisine.

Utilisé par

Wikipedia, The Guardian, New York Times, Drupal (intégration native), Reddit — tous les sites à fort trafic avec beaucoup de contenu statique ou semi-statique.

Comment Varnish fonctionne

Sans Varnish :
Client → Nginx/Apache → PHP/Node → Base de données → Réponse
         (chaque requête repart jusqu'à la BDD)

Avec Varnish :
Client → Varnish → [cache HIT]  → Réponse en ~1ms (depuis RAM)
                 → [cache MISS] → Nginx/Apache → PHP → BDD → Stocke en cache → Réponse
                   (seulement la 1ère fois ou quand le cache expire)

VCL — Varnish Configuration Language

Varnish est configurable via un langage propre, le VCL, qui permet de définir des règles de cache très précises :

vcl 4.0;

backend default {
    .host = "127.0.0.1";
    .port = "8080"; // Votre serveur app
}

sub vcl_recv {
    // Ne pas cacher les requêtes authentifiées
    if (req.http.Authorization || req.http.Cookie ~ "session=") {
        return (pass);
    }

    // Ne pas cacher les méthodes non-GET
    if (req.method != "GET" && req.method != "HEAD") {
        return (pass);
    }
}

sub vcl_backend_response {
    // Cacher pendant 1 heure les pages HTML
    if (beresp.http.Content-Type ~ "text/html") {
        set beresp.ttl = 1h;
    }

    // Cacher 1 semaine les assets statiques
    if (req.url ~ "\.(css|js|png|jpg|woff2)$") {
        set beresp.ttl = 7d;
        unset beresp.http.Set-Cookie;
    }
}

Métriques de performance

  • Hit rate : pourcentage de requêtes servies depuis le cache (objectif > 80%)
  • Backend load : charge sur le serveur d'application (doit chuter avec Varnish)
  • varnishstat : outil en temps réel pour monitorer les hits/misses
  • varnishlog : logs détaillés de chaque requête traitée
