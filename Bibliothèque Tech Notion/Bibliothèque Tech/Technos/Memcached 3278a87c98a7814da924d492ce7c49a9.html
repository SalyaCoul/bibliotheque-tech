Memcached

Statut
Terminé
TypeGestion de cache

🎯 En bref

Memcached est un système de cache distribué ultra-simple : il stocke des paires clé-valeur en mémoire RAM pour éviter d'interroger la base de données à chaque requête. Plus limité que Redis, mais extrêmement rapide pour des cas d'usage simples.

Analogie

C'est comme un Post-it géant sur votre bureau : au lieu de chercher un document dans une armoire (BDD), vous collez le résultat sur un papier visible immédiatement — et si le bureau est plein, vous jetez les Post-its les plus vieux.

Utilisé par

Facebook, Wikipedia, Twitter pour absorber des centaines de millions de requêtes par seconde.

Concepts clés

ConceptDescription
Clé-valeurStockage simple : une clé string → une valeur (string uniquement, pas de structures complexes)
TTLTime To Live : expiration automatique des données après N secondes
LRULeast Recently Used : éviction automatique des données les moins utilisées quand la mémoire est pleine
Multi-threadedTire parti de plusieurs cœurs CPU (avantage face à Redis single-thread)
DistribuéPlusieurs serveurs forment un pool — la répartition est gérée côté client

Exemple d'utilisation (PHP)

$memcached = new Memcached();
$memcached->addServer('localhost', 11211);

// Stocker une valeur pendant 1 heure (3600 secondes)
$memcached->set('user_42', json_encode($userData), 3600);

// Récupérer (pattern Cache-Aside)
$user = json_decode($memcached->get('user_42'), true);

if (!$user) {
    // Cache miss → requête BDD + mise en cache
    $user = $db->query('SELECT * FROM users WHERE id = 42');
    $memcached->set('user_42', json_encode($user), 3600);
}

Comparaison Memcached vs Redis

CritèreMemcachedRedis
Structures de donnéesString uniquementString, List, Set, Hash, Sorted Set, Stream...
Persistance❌ Non✅ Optionnelle (RDB/AOF)
Multi-threading✅ Oui❌ Single-thread (I/O async)
Pub/Sub❌ Non✅ Oui
Transactions❌ Non✅ MULTI/EXEC
Cas d'usageCache simple et ultra-rapideCache + pub/sub + sessions + files

Quand choisir Memcached ?

  • Cache de résultats de requêtes BDD simples
  • Données sans besoin de persistance
  • Charge très élevée nécessitant le multi-threading
  • Infrastructure déjà en place, pas besoin des features avancées de Redis
