🎯 En bref : REST et GraphQL sont les deux standards les plus connus, mais le monde des APIs est bien plus large. Selon le contexte (performance, temps réel, compatibilité historique...), d'autres types d'APIs entrent en jeu.

Analogie : Il existe plusieurs façons de communiquer — l'email, le SMS, le téléphone, la visioconférence... Chaque moyen de communication a ses avantages selon le contexte. Les types d'APIs, c'est pareil : REST pour la simplicité, WebSocket pour le temps réel, gRPC pour la vitesse, SOAP pour les systèmes d'entreprise legacy...

Autres types d'APIs

🔄 SOAP (Simple Object Access Protocol)

Le "vieux standard" d'entreprise. Utilise XML pour structurer les messages et fonctionne sur HTTP ou d'autres protocoles.

  • Points forts : Très strict, sécurisé, avec des standards de sécurité éprouvés (WS-Security). Parfait pour les systèmes bancaires et gouvernementaux.
  • Points faibles : Verbeux (les messages XML sont lourds), complexe à implémenter.
  • Utilisé par : Banques, assurances, systèmes gouvernementaux, ERP d'entreprise (SAP...).

⚡ gRPC (Google Remote Procedure Calls)

Créé par Google. Utilise HTTP/2 et des messages binaires (Protocol Buffers) au lieu de JSON.

  • Points forts : Extrêmement rapide (jusqu'à 10x plus rapide que REST/JSON), idéal pour la communication entre microservices.
  • Points faibles : Moins lisible que JSON, moins adapté aux navigateurs web directement.
  • Utilisé par : Google, Netflix, Cloudflare — surtout pour la communication interne entre services back-end.
  • Analogie : Si REST/JSON est comme des lettres (lisibles mais volumineuses), gRPC est comme un code morse ultra-optimisé — incompréhensible à l'œil nu mais ultra-rapide à transmettre.

🔌 WebSocket

Pas vraiment une "API" au sens traditionnel, mais un protocole de communication bidirectionnel et persistant.

  • Points forts : Connexion permanente ouverte entre client et serveur — les deux peuvent s'envoyer des messages à tout moment.
  • Utilisé pour : Chat en temps réel (Slack, Discord), cours de bourse en direct, jeux multijoueurs, notifications live, tableaux de bord temps réel.
  • Analogie : Alors qu'une API REST c'est un appel téléphonique (vous appelez, on répond, on raccroche), WebSocket c'est une conversation en continu où les deux parties peuvent parler librement à tout moment.

📋 OpenAPI / Swagger

Pas un type d'API en soi, mais un standard de documentation pour les APIs REST.

  • Objectif : Décrire une API REST de façon standardisée (format JSON ou YAML), pour générer automatiquement de la documentation, des clients, et faciliter les tests.
  • Swagger UI : Génère une interface web interactive à partir d'une spec OpenAPI — on peut tester l'API directement depuis le navigateur.
  • Utilisé par : Quasi toutes les grandes APIs publiques (Stripe, Twilio, GitHub...).

🏗️ JSON-RPC et XML-RPC

Des protocoles plus anciens qui exposent des "procédures" (fonctions) plutôt que des "ressources".

  • JSON-RPC : Version moderne, utilise JSON. Encore utilisé dans des contextes spécifiques (ex : nodes blockchain comme Ethereum).
  • XML-RPC : Précurseur de SOAP, presque plus utilisé aujourd'hui.

🗄️ APIs de bases de données

Permettent à une application de communiquer directement avec une base de données via SQL ou un langage spécifique.

  • Exemples : JDBC (Java), PDO (PHP), SQLAlchemy (Python)
  • Différence avec REST : Ces APIs fonctionnent directement entre le code backend et la BDD — elles ne sont pas exposées à l'extérieur.

📡 APIs de streaming (Server-Sent Events)

Variante légère du WebSocket où seul le serveur pousse des données vers le client (sens unique).

  • Utilisé pour : Notifications push, flux d'actualités, logs en temps réel.
  • Avantage vs WebSocket : Plus simple à mettre en place quand seul le serveur a besoin d'envoyer des données.
💡 En pratique, la grande majorité des projets web utilisent REST ou GraphQL. Les autres types entrent en jeu pour des besoins spécifiques : temps réel (WebSocket), performance inter-services (gRPC), ou compatibilité avec des systèmes legacy (SOAP).
