BDD relationnelles

Statut
Terminé
TypeBases de données

🎯 En bref

Les bases de données relationnelles organisent les données en tables liées entre elles via des clés. Le langage SQL permet de les interroger, filtrer, regrouper et joindre. C'est le modèle dominant depuis les années 70, fondé sur la théorie des ensembles et les transactions ACID.

Analogie

Une BDD relationnelle c'est comme un classeur Excel avec plusieurs feuilles qui se "connaissent" : la feuille "Clients" et la feuille "Commandes" sont reliées par un identifiant client. Un JOIN en SQL, c'est exactement ça : croiser deux feuilles sur une colonne commune.

Utilisé par

Pratiquement toutes les applications web : GitHub (MySQL), Airbnb (MySQL), Instagram (PostgreSQL), Shopify (MySQL), la majorité des backends PHP/Laravel/Symfony, Django, Rails.

Modèle relationnel — Exemple

TABLE users                    TABLE orders
┌────┬──────────┬──────────┐   ┌────┬──────────┬──────────┬────────┐
│ id │ name     │ email    │   │ id │ user_id  │ total    │ status │
├────┼──────────┼──────────┤   ├────┼──────────┼──────────┼────────┤
│  1 │ Alice    │ a@...    │   │  1 │    1     │  49.99   │ paid   │
│  2 │ Bob      │ b@...    │   │  2 │    1     │  12.50   │ pending│
│  3 │ Charlie  │ c@...    │   │  3 │    2     │  89.00   │ paid   │
└────┴──────────┴──────────┘   └────┴──────────┴──────────┴────────┘
                        users.id ←→ orders.user_id (clé étrangère)

SQL — Les requêtes essentielles

-- Sélectionner avec filtre
SELECT name, email FROM users WHERE email LIKE '%@gmail.com';

-- JOIN : croiser deux tables
SELECT u.name, o.total, o.status
FROM users u
INNER JOIN orders o ON u.id = o.user_id
WHERE o.status = 'paid';

-- Agrégation : total des commandes par client
SELECT u.name, COUNT(o.id) AS nb_commandes, SUM(o.total) AS total
FROM users u
LEFT JOIN orders o ON u.id = o.user_id
GROUP BY u.id, u.name
ORDER BY total DESC;

-- Transaction ACID : tout ou rien
BEGIN;
    UPDATE accounts SET balance = balance - 100 WHERE id = 1;
    UPDATE accounts SET balance = balance + 100 WHERE id = 2;
COMMIT; -- Ou ROLLBACK si erreur

Propriétés ACID

PropriétéSignificationExemple
AtomicitéTout ou rien : la transaction réussit complètement ou pas du toutVirement bancaire
CohérenceLa BDD passe d'un état valide à un autre état valideContraintes respectées
IsolationLes transactions concurrentes ne s'interfèrent pasPas de lecture sale
DurabilitéUne transaction validée persiste même en cas de crashDonnées sur disque

Quand utiliser une BDD relationnelle ?

  • Données structurées avec relations bien définies
  • Besoin de transactions ACID (finance, e-commerce, santé)
  • Requêtes complexes avec JOINs multiples
  • Équipes ayant des compétences SQL existantes
