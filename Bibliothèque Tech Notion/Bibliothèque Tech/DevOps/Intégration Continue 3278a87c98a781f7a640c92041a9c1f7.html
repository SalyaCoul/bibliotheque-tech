Intégration Continue

TypeCI/CD
État
Terminé

🎯 En bref

L'intégration continue (CI) est une pratique DevOps qui consiste à fusionner régulièrement le code de tous les développeurs dans un dépôt partagé, plusieurs fois par jour. À chaque fusion, une série de vérifications automatiques (compilation, tests, analyse de code) s'exécute pour détecter les problèmes immédiatement.

Analogie

C'est comme construire une maison avec plusieurs équipes en même temps : plutôt que d'attendre que tout le monde ait fini pour assembler les pièces (et découvrir que rien ne s'emboîte), chaque équipe livre ses briques chaque jour et on vérifie l'assemblage en temps réel. Les problèmes se règlent petit à petit, pas en catastrophe à la fin.

Utilisé par

Google, Netflix, Amazon, toutes les équipes de développement modernes — la CI est aujourd'hui le standard incontournable de tout projet professionnel.

Le flux CI typique

Développeur pousse du code
         ↓
   [Déclenchement automatique]
         ↓
┌─────────────────────────────────┐
│  1. Récupération du code (git)  │
│  2. Installation des dépendances│
│  3. Compilation / build         │
│  4. Tests unitaires             │
│  5. Tests d'intégration         │
│  6. Analyse statique (lint)     │
│  7. Scan de sécurité            │
│  8. Rapport de couverture       │
└─────────────────────────────────┘
         ↓
  ✅ Pipeline vert → code OK
  ❌ Pipeline rouge → notification immédiate

Concepts clés

ConceptDescription
PipelineSéquence d'étapes automatisées déclenchée par un événement (push, PR)
JobUnité d'exécution au sein du pipeline (ex : "run-tests", "build-docker")
ArtefactFichier produit par le pipeline (binaire, rapport, image Docker)
DéclencheurÉvénement qui lance le pipeline (push, PR, cron, tag)
EnvironnementContexte d'exécution (Docker, VM, runner cloud)
CacheConservation des dépendances entre pipelines pour accélérer

Exemple GitHub Actions

# .github/workflows/ci.yml
name: CI

on:
  push:
    branches: [main, develop]
  pull_request:

jobs:
  test:
    runs-on: ubuntu-latest

    steps:
      - uses: actions/checkout@v4

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'npm'

      - name: Install dependencies
        run: npm ci

      - name: Run linter
        run: npm run lint

      - name: Run tests
        run: npm test -- --coverage

      - name: Upload coverage
        uses: codecov/codecov-action@v3

CI vs CD

PratiqueSignificationObjectif
CI — Intégration ContinueMerger et valider le code automatiquementDétecter les bugs tôt
CD — Livraison ContinuePréparer automatiquement le déploiementToujours avoir un artefact déployable
CD — Déploiement ContinuDéployer automatiquement en productionLivrer en continu sans intervention manuelle

Bonnes pratiques

  • Un pipeline rapide (< 10 min) est un pipeline utilisé — optimiser avec le cache et la parallélisation
  • Tout commit sur la branche principale doit déclencher un pipeline
  • Les tests en échec bloquent le merge — jamais de "je merge quand même"
  • Garder les pipelines reproductibles : même résultat en local et en CI
