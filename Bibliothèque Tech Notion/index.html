👨‍💻 Métiers du Développement

💡 Pour bien comprendre : Une application web ou mobile, c'est comme un restaurant. Il y a la salle (ce que vous voyez), la cuisine (ce qui se passe en coulisses), et le système de gestion (la data). Les développeurs construisent tout ça !

🖥️ Développeur Backend

🎯 En bref : Le développeur backend construit la "cuisine" invisible d'une application — tout ce qui se passe en arrière-plan, loin des yeux de l'utilisateur.

Imaginez un restaurant : le serveur prend votre commande et vous apporte votre plat, mais c'est le cuisinier qui prépare tout en cuisine. Le développeur backend, c'est ce cuisinier. Il écrit le code qui tourne sur des serveurs — des ordinateurs puissants connectés à Internet — et qui gère toute la logique et les données de l'application.

Exemple concret : Quand vous vous connectez à Netflix et que votre liste de films apparaît, c'est le backend qui a vérifié vos identifiants, récupéré vos préférences et renvoyé les bons contenus.

En général, ce rôle implique :

  • Gérer les données : comment stocker et retrouver les informations (les comptes utilisateurs, les commandes d'un e-commerce...)
  • Écrire la logique métier : les règles de fonctionnement (ex : "un utilisateur non abonné ne peut pas accéder au contenu premium")
  • Construire des APIs : des "portes de communication" que le frontend utilise pour interroger le serveur

⚙️ Compétences et Technologies

Les langages les plus populaires pour un Développeur Backend sont PHP, Node.js, Python, Java et Ruby

Les frameworks les plus populaires pour ces langages sont :

  • Laravel et Symfony pour PHP
  • Express.js et NestJS pour Node.js
  • Django et Flask pour Python
  • Spring pour Java

Les développeurs Backend ont aussi besoin d’avoir :

  • Une compréhension de technos telles que les bases de données (MySQL, ou PostgreSQL par exemple), les caches (Redis ou Memcached), les files d’attente des messages afin de manipuler la data.
  • Une compréhension des algorithmes et des structures de données afin d’écrire de bonnes APIs et d’avoir une logique business
  • Un peu de connaissances DevOps (infra cloud, CI/CD) pour bien déployer le code

🎨 Développeur Frontend

🎯 En bref : Le développeur frontend construit tout ce que vous voyez et avec quoi vous interagissez dans une application : boutons, menus, pages, animations.

Si le backend est la cuisine, le frontend est la salle du restaurant — c'est ce que le client voit et touche. Quand vous cliquez sur un bouton, faites défiler une page ou remplissez un formulaire, c'est le frontend qui gère tout ça.

Exemples concrets :

  • La page d'accueil de votre réseau social avec son fil d'actualité → frontend
  • Le menu déroulant qui apparaît quand vous cliquez sur votre profil → frontend
  • L'animation de chargement pendant qu'une page se charge → frontend

⚙️ Compétences techniques

Les technologies de base du frontend sont :

  • HTML : le squelette de la page (structure du contenu — titres, paragraphes, images...)
  • CSS : l'habillage (couleurs, polices, mise en page, animations)
  • JavaScript : l'interactivité (boutons cliquables, formulaires dynamiques, contenus qui se mettent à jour sans recharger la page)

Les frameworks JavaScript populaires sont React, Vue.js et Angular — ce sont des boîtes à outils qui accélèrent la création d'interfaces complexes. React est de loin le plus utilisé aujourd'hui.

Sur des applications très sophistiquées, le développeur frontend peut aussi gérer l'architecture applicative : comment les données circulent dans l'interface, comment la page reste rapide même avec beaucoup de contenu...

📐 Principes de conception de logiciels

💡 Ces principes sont les "règles d'or" que les développeurs suivent pour écrire du code solide, facile à faire évoluer et à maintenir. Pensez-y comme aux règles d'architecture pour construire un bâtiment durable plutôt qu'une maison en carton.

Principes SOLID

🎯 En bref : SOLID, c'est 5 règles de bonne conduite pour écrire du code propre, facile à faire évoluer et à maintenir. Imaginez des LEGO : SOLID, c'est le manuel pour assembler les pièces de manière à pouvoir facilement en ajouter ou en changer sans tout démonter.

En programmation orientée objet, SOLID est un acronyme regroupant 5 principes de conception destinés à produire des architectures logicielles plus compréhensibles, flexibles et maintenables. Les principes sont un sous-ensemble de nombreux principes promus par l'ingénieur logiciel et instructeur américain Robert C. Martin.

Les cinq principes sont :

  • S — Single Responsibility (Responsabilité unique) : chaque partie du code ne fait qu'une seule chose. Exemple : une fonction qui calcule un prix ne doit pas aussi envoyer l'email de confirmation.
  • O — Open/Closed (Ouvert/Fermé) : on peut ajouter de nouvelles fonctionnalités sans modifier ce qui existe déjà. Exemple : ajouter PayPal sans toucher au code du paiement par carte.
  • L — Liskov Substitution (Substitution de Liskov) : une version spéciale d'un objet doit pouvoir remplacer l'original sans casser le programme. Exemple : si "Pingouin" hérite de "Oiseau", le remplacer par un Oiseau ne doit pas provoquer d'erreur.
  • I — Interface Segregation (Ségrégation des interfaces) : mieux vaut plusieurs petits contrats spécialisés qu'un seul gros contrat fourre-tout. Exemple : ne pas forcer un oiseau à avoir une méthode "nager" juste parce qu'il hérite de "Animal".
  • D — Dependency Inversion (Inversion des dépendances) : le code doit dépendre de concepts généraux, pas de détails spécifiques. Exemple : votre app ne devrait pas dépendre de "MySQL" en particulier, mais d'un concept générique de "base de données" — pour pouvoir en changer facilement si besoin.

Le Domain Driven Design

🎯 En bref : Le DDD, c'est l'idée que le code doit parler le même langage que le métier qu'il représente. Exemple : pour une banque, les classes du code s'appelleront "Compte", "Virement", "Découvert" — pas des termes techniques incompréhensibles.

Le domain driven design (conception pilotée par le domaine) est un principe selon lequel la conception d'une application toute entière doit être définie sur le design du domaine. C’est-à-dire qu’il faut représenter le métier directement dans le code, plutôt que de coder le métier.

C’est une philosophie de conception, on se base sur le métier pour assembler la solution que l’on apporte au business, autrement dit le code.

Ainsi, la structure, le nom des classes, des champs, les actions des fonctions doivent refléter le métier.

On dit qu'une personne du métier doit presque être capable de comprendre l'intention du code en le lisant — c'est avant tout la compréhension du métier par l'ensemble de l'équipe de développement.

Exemple concret : Pour un e-commerce, le code contiendra des concepts comme "Panier", "Commande", "Livraison", "Retour" — des mots que le responsable produit utilise tous les jours.

L’injection de dépendance

🎯 En bref : Plutôt que chaque partie du code crée elle-même les outils dont elle a besoin, on lui fournit ces outils de l'extérieur. Analogie : un cuisinier à qui on remet ses couteaux en main propre, plutôt qu'il aille les chercher lui-même — plus flexible, plus facile à changer.

L'injection de dépendance (DI, pour Dependency Injection en anglais) est un concept de conception logicielle visant à réduire les dépendances entre les composants d'une application. Elle est utilisée pour rendre les systèmes logiciels plus modulaires, flexibles et faciles à tester. Voici une explication plus détaillée de l'injection de dépendance :

  1. Définition :
    • L'injection de dépendance est un principe selon lequel les dépendances d'un objet sont fournies par une entité externe plutôt que d'être créées à l'intérieur de l'objet lui-même. Au lieu d'une classe créant ses propres instances d'autres classes dont elle dépend, ces dépendances sont "injectées" depuis l'extérieur.
  1. Objectif :
    • L'objectif principal de l'injection de dépendance est de réduire le couplage entre les différentes parties d'un système logiciel. Un couplage réduit signifie que les composants d'un système sont moins dépendants les uns des autres, ce qui facilite la maintenance, la modification et l'extension du code.
  1. Composants Principaux :
    • Service (Dépendance) : C'est la classe ou l'objet dont une autre classe dépend.
    • Client : C'est la classe ou l'objet qui a besoin du service (dépendant du service).
  1. Types d'Injection de Dépendance :
    • Constructeur : Les dépendances sont injectées via le constructeur de la classe.
    • Setter : Les dépendances sont injectées via des méthodes de type setter.
    • Interface : Les dépendances sont injectées via des interfaces, où la classe cliente dépend d'une interface plutôt que d'une implémentation concrète.
  1. Avantages :
    • Réutilisabilité : Les composants peuvent être plus facilement réutilisés car ils ne sont pas fortement couplés à des détails d'implémentation spécifiques.
    • Testabilité : Il est plus facile de tester les composants individuellement, car les dépendances peuvent être remplacées par des versions simulées (mocks) ou des objets de test.
    • Maintenabilité : Les changements dans une dépendance n'affectent pas nécessairement toutes les classes qui l'utilisent, réduisant ainsi l'impact des modifications.

Design Patterns

🎯 En bref : Les design patterns sont des recettes de cuisine éprouvées pour résoudre des problèmes courants en programmation. Tout comme un chef ne réinvente pas chaque jour la sauce béchamel, un développeur s'appuie sur ces "recettes" reconnues pour gagner du temps et éviter les erreurs.

Les design patterns, ou patrons de conception en français, sont des solutions standards reconnues comme bonnes pratiques pour résoudre un problème de conception d'un logiciel/site web/application.

Ils sont issus de l'expérience de développeurs ayant rencontré des problèmes récurrents et ayant mis en place une façon standard de les résoudre.

Le design pattern donne une direction, une façon d'organiser les choses dans le code.

Pour un développeur, les connaître lui permet d'avoir des concepts de solutions fiables et testées pour tout type de problème.

Les design patterns ont formellement été reconnus suite à la parution du livre Design Patterns: Elements of Reusable Software de la “bande des 4” (The Gang of Four).

Ils décrivent 23 design patterns, découpés en 3 catégories :

  • Les patrons de création : comment créer des objets de manière intelligente et flexible (ex : s'assurer qu'un seul objet de configuration existe dans toute l'application)
  • Les patrons structurels : comment organiser les relations entre les composants d'un système (ex : faire fonctionner ensemble deux composants incompatibles grâce à un adaptateur)
  • Les patrons comportementaux : comment les composants interagissent et se coordonnent (ex : notifier automatiquement plusieurs éléments quand une donnée change)

⚠️ À noter : Les design patterns sont des concepts à comprendre, pas du code à copier-coller. Mal utilisés, ils peuvent complexifier inutilement des choses simples. L'objectif est de saisir la logique derrière chaque pattern, pas de l'appliquer mécaniquement.

Exemples de Design Pattern

  1. Singleton — "À n'en vouloir qu'un"
    • Assure qu'une seule instance d'un objet existe dans tout le programme
    • Exemple : les paramètres de configuration de l'app (connexion BDD, clés API...) — une seule instance suffit pour tout le monde
  1. Factory Method — "La chaîne de fabrication"
    • Définit comment créer un objet, mais laisse les sous-classes décider lequel créer exactement
    • Exemple : un système de notifications qui crée des alertes email, SMS ou push selon le contexte
  1. Abstract Factory — "L'usine d'usines"
    • Crée des familles d'objets assortis sans préciser leurs classes exactes
    • Exemple : un thème d'interface (clair/sombre) qui génère en cohérence les boutons, menus et fenêtres assortis
  1. Builder — "La construction brique par brique"
    • Construit un objet complexe étape par étape
    • Exemple : générer un rapport PDF avec des sections optionnelles (introduction, graphiques, conclusion)
  1. Prototype — "Le photocopieur"
    • Crée de nouveaux objets en copiant un modèle existant
    • Exemple : dans un jeu vidéo, dupliquer un personnage ennemi pour créer rapidement 50 ennemis similaires
  1. Adapter — "L'adaptateur de prise électrique"
    • Permet à des composants incompatibles de travailler ensemble
    • Exemple : brancher une ancienne bibliothèque dans un nouveau framework grâce à une couche d'adaptation
  1. Decorator — "Les couches supplémentaires"
    • Ajoute des fonctionnalités à un objet sans modifier son code
    • Exemple : ajouter de la compression puis du chiffrement à un fichier, couche par couche
  1. Observer — "Le système d'alertes"
    • Quand un objet change, tous ceux qui le "surveillent" sont automatiquement notifiés
    • Exemple : dans une app météo, quand la température change, le widget, les alertes et le graphique se mettent tous à jour automatiquement
  1. Strategy — "Choisir son approche à la volée"
    • Permet de changer facilement la manière de faire quelque chose selon le contexte
    • Exemple : un e-commerce qui applique différentes règles de réduction selon le type de client (VIP, nouveau, code promo...)
  1. Command — "Le bon de commande"
    • Encapsule une action dans un objet, ce qui permet de l'annuler, la rejouer ou la mettre en file d'attente
    • Exemple : le Ctrl+Z dans un éditeur de texte — chaque action est un "command" qu'on peut défaire

🔄 Méthodes de développement

💡 Ce sont les approches et philosophies que les équipes tech adoptent pour organiser leur travail et livrer des logiciels de qualité.

Behavior driven development (BDD)

🎯 En bref : Le BDD, c'est développer en partant du comportement attendu par l'utilisateur, décrit dans un langage que tout le monde comprend — pas seulement les développeurs.

Le problème que ça résout : Souvent, les développeurs construisent ce qu'ils comprennent des besoins, et les non-techniques découvrent à la fin que ce n'est pas tout à fait ce qu'ils voulaient. Le BDD évite ce décalage.

Le Behavior Driven Development (BDD) est une méthodologie de développement logiciel qui met l'accent sur la collaboration entre tous les membres de l'équipe — développeurs, testeurs et responsables produit. L'objectif principal est de créer un langage commun compréhensible par tous pour définir le comportement attendu du système.

Voici quelques points clés du BDD :

  1. Scénarios Utilisateurs : Le BDD commence par la création de scénarios utilisateurs qui décrivent le comportement souhaité du système du point de vue de l'utilisateur. Ces scénarios sont souvent rédigés dans un langage naturel compréhensible par tous.
  1. Langage Gherkin : Pour formaliser les scénarios, le BDD utilise le langage Gherkin, simple et lisible par tous, avec des mots-clés : "Étant donné que..." (contexte), "Quand..." (action de l'utilisateur), "Alors..." (résultat attendu). Exemple : "Étant donné qu'un utilisateur est connecté, Quand il ajoute un article au panier, Alors le compteur du panier s'incrémente de 1."
  1. Collaboration : Le BDD encourage la collaboration entre les différentes parties prenantes dès le début du processus de développement. Les membres de l'équipe travaillent ensemble pour définir et comprendre les scénarios.
  1. Automatisation des Tests : Les scénarios écrits en langage Gherkin peuvent être automatisés pour vérifier que le système se comporte comme spécifié. Des outils tels que Cucumber, Behave, SpecFlow, entre autres, permettent de lier ces scénarios à des tests automatisés.
  1. Réduction des Ambiguïtés : Le BDD aide à réduire les ambiguïtés dans les exigences en les exprimant de manière claire et compréhensible. Cela améliore la communication au sein de l'équipe et réduit les risques d'interprétation erronée.

En adoptant le BDD, les équipes peuvent améliorer la communication, renforcer la compréhension commune des fonctionnalités à développer, et créer des tests automatisés qui reflètent les scénarios d'utilisation réels. Cela contribue à livrer des logiciels de meilleure qualité, alignés sur les attentes des utilisateurs et des parties prenantes.

Test driven development

🎯 En bref : Le TDD, c'est écrire les tests avant d'écrire le code. Ça peut sembler contre-intuitif, mais c'est très efficace ! C'est comme écrire la correction d'un exercice avant l'exercice lui-même — ça oblige à réfléchir précisément à ce qu'on attend.

Le Test Driven Development (TDD) est une approche où les tests automatisés sont écrits avant le code de production. L'objectif est de s'assurer que chaque composant fonctionne exactement comme prévu.

Le cycle TDD se résume en 3 étapes (aussi appelé "Red-Green-Refactor") :

  1. 🔴 Rouge : J'écris un test qui décrit ce que je veux → il échoue (le code n'existe pas encore)
  1. 🟢 Vert : J'écris le minimum de code pour que le test passe
  1. 🔵 Refacto : Je nettoie et améliore le code sans changer son comportement, puis je recommence

Exemple : avant d'écrire une fonction de réduction, j'écris le test : "si commande > 100€, réduction = 10%". Ce test échoue → j'écris la fonction → le test passe → je nettoie le code.

TDD et BDD sont complémentaires dans la mesure où TDD fournit une base solide de tests unitaires, tandis que BDD élargit la portée en incluant des tests de comportement à un niveau plus élevé. Les scénarios BDD peuvent souvent être utilisés pour guider le processus TDD en identifiant les fonctionnalités à implémenter et les tests unitaires à écrire.

Serverless

🎯 En bref : Le serverless permet aux développeurs de déployer du code dans le cloud sans se préoccuper de gérer des serveurs.

Analogie : C'est comme utiliser un taxi plutôt qu'avoir sa propre voiture. Vous n'avez pas à vous occuper de l'entretien ou de l'assurance — vous appelez quand vous en avez besoin, vous payez la course, c'est tout.

Le serverless (ou "computing sans serveur") est un modèle dans lequel les développeurs construisent des applications sans se soucier de la gestion directe des serveurs.

1. Abstraction des Serveurs :

  • Les développeurs n'ont pas à configurer ni maintenir des serveurs. Tout ça est géré par le fournisseur cloud en coulisses.

2. Fonctionnement par Événements :

  • Les applications serverless sont basées sur des fonctions (appelées aussi "fonctions serverless" ou "fonctions sans serveur"). Ces fonctions sont déclenchées par des événements tels que des requêtes HTTP, des mises à jour de base de données, des modifications de fichiers, etc.

3. Élasticité Automatique :

  • Le modèle serverless offre une élasticité automatique. Les fonctions sont exécutées à la demande en réponse aux événements. L'échelle automatique signifie que les ressources sont provisionnées et déprovisionnées dynamiquement en fonction du volume de demandes.

4. Paiement à l'Usage :

  • Les utilisateurs ne paient que pour le temps réel d'exécution de leurs fonctions. Il n'y a pas de coûts fixes associés à la maintenance de serveurs inactifs.

5. Gestion des Infrastructures Côté Fournisseur :

  • Les fournisseurs de cloud, tels que AWS Lambda, Azure Functions, et Google Cloud Functions, gèrent l'infrastructure sous-jacente. Les développeurs se concentrent sur l'écriture du code métier.

6. Rapidité de Déploiement :

  • Les développeurs peuvent déployer rapidement des fonctions individuelles sans se soucier de l'ensemble de l'infrastructure. Cela favorise un développement agile et des mises à jour fréquentes.

7. Utilisations Courantes :

  • Le serverless est souvent utilisé pour des scénarios tels que les microservices, les applications backend sans serveur, le traitement de données en temps réel, et la création d'API.

8. Limitations :

  • Bien que le serverless offre de nombreux avantages, il peut ne pas convenir à toutes les applications. Certaines applications nécessitant un état persistant ou un temps d'exécution prolongé peuvent trouver des limitations dans ce modèle.

🧪 Testing — Les Tests Logiciels

🎯 En bref : Tester un logiciel, c'est s'assurer qu'il fait bien ce qu'il est censé faire — et qu'il ne fait pas ce qu'il ne devrait pas faire.

Imaginez que vous faites livrer une voiture sur mesure. Avant la livraison, le constructeur la teste : le moteur démarre-t-il ? Les freins fonctionnent-ils ? La climatisation marche-t-elle ? Les tests logiciels, c'est exactement la même démarche, mais pour du code.

Les tests permettent de :

  1. Vérifier que le logiciel implémente correctement chaque fonction spécifiée ("Est-ce que ça fait ce qui était prévu ?")
  1. Valider que le logiciel correspond bien aux attentes du client ("Est-ce que ça répond au vrai besoin ?")

Les tests fonctionnels

Les tests fonctionnels vérifient que chaque fonctionnalité de l'application fonctionne conformément aux spécifications.

En clair : on fournit une entrée à la fonctionnalité, et on vérifie que la sortie correspond à ce qu'on attendait.

Exemple : tester que lorsqu'un utilisateur clique sur "Commander", sa commande est bien enregistrée et un email de confirmation est envoyé.

Ces tests couvrent l'interface utilisateur, les APIs, la base de données, la sécurité... Ils peuvent être réalisés manuellement (un testeur clique) ou automatiquement (un script exécute les actions à la place du testeur).

Le rôle du QA Tester

QA Tester (Tests Manuels) :

  • Rôle : Un QA Tester, spécialisé dans les tests manuels, joue un rôle crucial dans la validation de la qualité d'un logiciel. Son objectif principal est de s'assurer que le logiciel répond aux exigences spécifiées et fonctionne correctement du point de vue de l'utilisateur.
  • Responsabilités :
    • Effectuer des tests manuels en suivant des scénarios de test prédéfinis.
    • Identifier, documenter et signaler les anomalies ou les problèmes de manière détaillée.
    • Participer aux phases de planification et de conception des tests.
    • Collaborer avec les membres de l'équipe pour comprendre les exigences et les fonctionnalités à tester.

Test Automation Engineer (Automatisation des Tests) :

  • Rôle : Un Test Automation Engineer se concentre sur l'automatisation des tests, utilisant des outils et des frameworks pour créer des suites de tests automatisées. L'objectif est d'accélérer le processus de test, d'améliorer la couverture des tests et de détecter rapidement les régressions.
  • Responsabilités :
    • Concevoir, développer et maintenir des scripts de test automatisés.
    • Identifier les scénarios de test adaptés à l'automatisation.
    • Intégrer les tests automatisés dans les pipelines CI/CD.
    • Collaborer avec les développeurs pour s'assurer que les nouvelles fonctionnalités sont bien testées.

Distinctions :

  • Les QA Testers manuels sont souvent impliqués dans les phases de test fonctionnel, de régression, et de validation manuelle. Ils ont une compréhension approfondie des exigences fonctionnelles et des scénarios d'utilisation.
  • Les Test Automation Engineers sont plus axés sur l'efficacité à long terme. Ils investissent du temps dans le développement et la maintenance des scripts automatisés, ce qui permet des cycles de test plus rapides, en particulier pour les tests de régression.

Collaboration :

  • Les deux rôles travaillent souvent en étroite collaboration. Les QA Testers peuvent aider à identifier les cas de test prioritaires pour l'automatisation, tandis que les Test Automation Engineers peuvent fournir un support technique aux QA Testers pour l'exécution des tests automatisés.

⚙️ DevOps

🎯 En bref : DevOps, c'est une façon de travailler qui rapproche les équipes de développement (qui écrivent le code) et d'exploitation (qui font tourner les serveurs), pour livrer des logiciels plus vite et de manière plus fiable.

Analogie : Avant DevOps, c'était comme une cuisine où les cuisiniers (devs) préparaient les plats et les passaient par un guichet aux serveurs (ops), qui se débrouillaient ensuite. Avec DevOps, tout le monde travaille main dans la main de bout en bout.

DevOps, une contraction de "Development" (Développement) et "Operations" (Exploitation), est une approche qui vise à améliorer la collaboration entre les équipes dev et ops. L'objectif : accélérer les livraisons, améliorer la fiabilité et la stabilité des systèmes.

Les grands principes de DevOps :

1. Collaboration et Communication :

  • DevOps met l'accent sur la collaboration étroite entre les équipes de développement, d'exploitation, et d'autres parties prenantes. La communication transparente et régulière est encouragée pour éviter les silos d'information.

2. Automatisation :

  • L'automatisation est au cœur de DevOps. Les tâches manuelles répétitives sont automatisées autant que possible, ce qui inclut l'automatisation des tests, du déploiement, de la configuration, et d'autres processus.

3. Livraison Continue :

  • DevOps encourage la livraison continue, où des changements de code sont livrés fréquemment, rapidement, et de manière fiable. Cela implique des pipelines d'intégration continue et de déploiement continu (CI/CD).

4. Infrastructure as Code (IaC) :

  • L'infrastructure (serveurs, réseaux, configurations) est décrite dans des fichiers texte comme du code — ce qui la rend reproductible et partageable. Analogie : avoir une recette précise plutôt que de devoir reconstruire de tête à chaque fois.

5. Surveillance et Gestion des Incidents :

  • La surveillance continue des applications et de l'infrastructure est essentielle. En cas d'incident, DevOps favorise une approche proactive pour diagnostiquer, résoudre, et prévenir les problèmes futurs.

6. Culture du Feedback :

  • DevOps favorise une culture du feedback, où les retours d'information sont recueillis à chaque étape du cycle de vie du développement. Cela inclut le feedback des utilisateurs, des tests automatisés, et des analyses de performances.

7. Agilité :

  • DevOps s'aligne souvent sur les principes de l'agilité en favorisant des cycles de développement courts, des itérations rapides, et une capacité à s'adapter rapidement aux changements.

8. Sécurité :

  • La sécurité est intégrée dès le départ dans le processus DevOps. Cela inclut des pratiques de sécurité telles que l'analyse statique et dynamique du code, la gestion des identités, et la surveillance de la sécurité.

9. Conteneurisation :

  • Les applications sont "empaquetées" dans des conteneurs (grâce à Docker). Un conteneur, c'est comme une boîte qui contient l'application et tout ce dont elle a besoin pour fonctionner — elle peut s'exécuter identiquement partout, que ce soit sur votre ordinateur ou sur un serveur à l'autre bout du monde.

10. Orchestration de Conteneurs :

  • Quand on a beaucoup de conteneurs à gérer, des outils comme Kubernetes les orchestrent automatiquement : ils démarrent de nouveaux conteneurs si la charge augmente, redémarrent ceux qui tombent en panne, etc.

11. Gestion de Registre de Conteneurs :

  • Les "images" (modèles) de conteneurs sont stockées dans des registres (comme Docker Hub), pour être distribuées et réutilisées facilement par toute l'équipe.

L'objectif global de DevOps est de créer une culture où les équipes travaillent de manière collaborative, où les processus sont automatisés pour favoriser la rapidité et la fiabilité, et où la qualité des logiciels est constamment améliorée. Cette approche contribue à surmonter les défis traditionnels de la séparation entre le développement et l'exploitation, permettant aux organisations de livrer des logiciels de manière plus rapide, plus fiable, et plus efficace.

Le Build

🎯 En bref : Le "build" est la phase où le code source écrit par les développeurs est transformé en quelque chose d'exécutable et prêt à être déployé.

Analogie : c'est comme transformer des ingrédients bruts (le code) en un plat cuisiné (le programme), prêt à être servi.

Le build inclut la compilation (traduction du code en langage machine), la gestion des dépendances (récupération des bibliothèques nécessaires), les tests automatisés et la création d'artefacts prêts à déployer (binaires, conteneurs...).

  1. Gestion de Code Source (Version Control):
    • Étape: Les développeurs collaborent et gèrent le code source à l'aide de systèmes de contrôle de version comme Git.
    • Technologies: Git, SVN.
  1. Intégration Continue (CI):
    • Étape: Les changements de code sont automatiquement intégrés dans un référentiel partagé plusieurs fois par jour.
    • Technologies: Jenkins, Travis CI, GitLab CI.
  1. Automatisation de Build:
    • Étape: Compilation du code source, gestion des dépendances, et création d'artefacts exécutables.
    • Technologies: Apache Maven, Gradle, npm.
  1. Tests Automatisés:
    • Étape: Exécution de tests automatisés pour garantir la qualité du code.
    • Technologies: JUnit, TestNG (pour Java), pytest (pour Python).
  1. Empaquetage et Artefacts:
    • Étape: Création de packages d'application et d'artefacts pour le déploiement ultérieur.
    • Technologies: Docker pour la conteneurisation, JFrog Artifactory.

Compétences Build en DevOps:

  • Connaissance approfondie des outils de gestion de version.
  • Compréhension de l'intégration continue et des pipelines CI/CD.
  • Maîtrise des outils de build et des langages de script.
  • Expérience dans la mise en place de tests automatisés.

Le Run

🎯 En bref : Le "run", c'est faire tourner l'application en production — s'assurer qu'elle est disponible, performante et stable pour les utilisateurs.

Analogie : si le build est la préparation du plat, le run c'est le service en salle — s'assurer que le plat arrive chaud, à temps, et que tous les clients sont servis.

Le run englobe le déploiement, la gestion de la configuration, l'orchestration des conteneurs, la surveillance des performances et la résolution des incidents.

  1. Gestion de Configuration:
    • Étape: Définition et gestion des configurations d'application et d'infrastructure.
    • Technologies: Ansible, Puppet, Chef.
  1. Déploiement Continu (CD):
    • Étape: Automatisation du déploiement des applications dans différents environnements.
    • Technologies: Jenkins, GitLab CI, Kubernetes.
  1. Orchestration de Conteneurs:
    • Étape: Gestion des conteneurs pour simplifier le déploiement et l'évolutivité.
    • Technologies: Kubernetes, Docker Swarm.
  1. Surveillance (Monitoring):
    • Étape: Surveillance en temps réel des performances et de la disponibilité des applications.
    • Technologies: Prometheus, Grafana, ELK Stack.
  1. Gestion des Logs:
    • Étape: Collecte et analyse des journaux pour le débogage et la surveillance.
    • Technologies: ELK Stack (Elasticsearch, Logstash, Kibana), Splunk.

Compétences Run en DevOps:

  • Expertise dans la gestion de la configuration et l'automatisation du déploiement.
  • Connaissance approfondie des orchestrateurs de conteneurs.
  • Expérience dans la mise en place de solutions de surveillance et de gestion des logs.

Approches dérivées du DevOps

DevSecOps

🎯 En bref : DevSecOps = DevOps + Sécurité intégrée dès le début, pas ajoutée à la fin.

Analogie : plutôt que de vérifier les serrures une fois la maison construite, on pense à la sécurité dès les plans de l'architecte.

DevSecOps est une approche qui intègre la sécurité (Sec) dès le début du cycle de développement et tout au long des opérations. C'est une évolution naturelle de DevOps qui rapproche les équipes dev, ops et sécurité pour créer un processus continu, rapide et sécurisé.

Voici quelques points clés concernant DevSecOps :

*1. Collaboration Continue :

  • DevSecOps encourage la collaboration continue entre les équipes de développement, d'exploitation et de sécurité. Au lieu de traiter la sécurité comme une étape distincte en fin de processus, elle est intégrée à chaque phase du développement.

*2. Automatisation de la Sécurité :

  • L'automatisation est une pierre angulaire de DevSecOps. Cela inclut l'automatisation des tests de sécurité, des analyses de vulnérabilité, des audits de code, etc. L'objectif est d'identifier et de corriger les problèmes de sécurité dès que possible.

*3. Éducation et Sensibilisation :

  • Les équipes de DevSecOps mettent l'accent sur l'éducation et la sensibilisation. Les développeurs sont formés aux meilleures pratiques de sécurité, et la culture de la sécurité est promue à tous les niveaux de l'organisation.

*4. Tests de Sécurité Continus :

  • Les tests de sécurité continus sont intégrés au processus de construction et de déploiement. Cela inclut des analyses statiques de sécurité, des tests dynamiques de sécurité, des tests d'intrusion, etc.

*5. Gestion des Identités et des Accès :

  • La gestion des identités et des accès est gérée de manière sécurisée. Les principes du moins de privilèges et du besoin de savoir sont appliqués pour minimiser les risques.

*6. Infrastructure en tant que Code Sécurisée :

  • La sécurité est également intégrée dans la gestion de l'infrastructure en tant que code (IaC). Les configurations de sécurité sont gérées de manière codifiée pour éviter les configurations incorrectes.

*7. Détection et Réponse Rapides :

  • En cas de détection d'une menace ou d'une violation de sécurité, DevSecOps favorise des réponses rapides. Cela peut inclure des correctifs automatisés, des mises à jour de sécurité, et des mécanismes de réponse aux incidents.

*8. Compliance Automatisée :

  • Les processus de compliance sont automatisés autant que possible pour s'assurer que les applications et les infrastructures respectent les normes de sécurité et les réglementations.

*9. Monitoring et Analyse des Logs :

  • Un suivi continu et une analyse des logs sont effectués pour détecter tout comportement suspect et réagir rapidement en cas d'incident de sécurité.

L'objectif global de DevSecOps est de créer une culture où la sécurité n'est pas un frein au développement, mais une partie intégrante du processus qui améliore la qualité globale de l'application tout en réduisant les risques de sécurité.

FinOps

FinOps, ou Gestion Financière des Opérations (Financial Operations), est une approche et une discipline émergente qui vise à optimiser les coûts dans le cloud et à améliorer la compréhension financière des opérations liées à l'informatique. L'objectif est de permettre aux organisations de mieux gérer leurs dépenses liées aux ressources informatiques, en particulier dans le contexte des services cloud.

Voici quelques points clés concernant le FinOps :

*1. Collaboration entre les Équipes Financières et Techniques :

  • FinOps encourage une collaboration étroite entre les équipes financières, les équipes techniques et les opérations informatiques. Il s'agit de créer une compréhension commune des coûts associés aux services cloud.

*2. Cadre de Gestion des Coûts :

  • Le FinOps fournit un cadre de gestion des coûts qui aide les organisations à comprendre, mesurer et optimiser les coûts liés aux services cloud. Cela inclut la budgétisation, l'analyse des coûts, et la prise de décision basée sur des données financières.

*3. Transparence Financière :

  • Une transparence financière accrue est un pilier du FinOps. Cela implique de fournir des informations détaillées sur les coûts aux équipes techniques, aux gestionnaires de projet et aux parties prenantes afin qu'ils comprennent l'impact financier de leurs décisions.

*4. Analyse des Coûts et des Performances :

  • FinOps combine l'analyse des coûts avec des métriques de performance. Il ne s'agit pas seulement de réduire les coûts, mais aussi de maximiser la valeur obtenue pour chaque dollar dépensé.

*5. Automatisation des Processus Financiers :

  • L'automatisation des processus financiers est un aspect clé du FinOps. Cela inclut l'automatisation des processus de facturation, la gestion des budgets, la surveillance des coûts, etc.

*6. Éducation et Sensibilisation :

  • FinOps encourage l'éducation et la sensibilisation des équipes techniques sur l'impact financier de leurs actions. Les équipes sont formées pour prendre des décisions qui optimisent les coûts tout en maintenant les performances.

*7. Optimisation Continue :

  • L'optimisation des coûts n'est pas une activité ponctuelle, mais un processus continu. Les équipes travaillent constamment à identifier et à mettre en œuvre des opportunités d'optimisation des coûts.

*8. Alignement avec les Objectifs Métier :

  • FinOps vise à aligner les dépenses informatiques avec les objectifs métier de l'organisation. Cela signifie que les investissements dans le cloud sont alignés sur la création de valeur pour l'entreprise.

En résumé, le FinOps est une approche holistique qui reconnaît que la gestion des coûts dans le cloud ne relève pas uniquement de la responsabilité des équipes financières, mais nécessite la collaboration de l'ensemble de l'organisation, en particulier des équipes techniques qui prennent des décisions directes impactant les coûts.

DataOps

DataOps, contraction de Data Operations, est une approche opérationnelle qui vise à améliorer la collaboration et l'efficacité dans le cycle de vie des données, de la collecte à l'analyse. C'est une évolution naturelle des pratiques DevOps, adaptée au contexte spécifique des données.

Voici quelques points clés concernant le DataOps :

  1. Collaboration Interfonctionnelle : DataOps encourage la collaboration étroite entre les équipes de développement, d'opérations, et les professionnels des données. L'objectif est de briser les silos traditionnels et de favoriser la communication pour améliorer le flux de travail des données.
  1. Automatisation : L'automatisation est essentielle en DataOps. Cela inclut l'automatisation des processus de collecte, de nettoyage, de transformation, de gestion et d'analyse des données. L'automatisation améliore la reproductibilité et réduit les erreurs manuelles.
  1. Versionnage des Données : Comme le code source, les données peuvent être versionnées. Le versionnage des données permet de suivre les changements au fil du temps, d'assurer la traçabilité et de faciliter la collaboration.
  1. Monitoring et Gestion des Performances : Le suivi continu des données et la gestion des performances sont intégrés pour garantir que les données sont disponibles, fiables et répondent aux exigences de qualité.
  1. Infrastructure en tant que Code (IaC) pour les Données : L'IaC est appliquée aux données, ce qui signifie que l'infrastructure nécessaire pour collecter, stocker et traiter les données est gérée de manière codifiée.
  1. Sécurité des Données : La sécurité des données est une préoccupation majeure en DataOps. Les politiques de sécurité sont intégrées dès le début, avec des contrôles d'accès appropriés et des mesures de protection des données.
  1. Catalogage et Métadonnées : Un catalogue de données centralisé et des métadonnées sont utilisés pour documenter, organiser et faciliter la recherche de données. Cela améliore la découverte des données et la gouvernance.
  1. Agilité : DataOps vise à rendre le processus de gestion des données plus agile, permettant des itérations rapides et une réactivité aux changements dans les besoins commerciaux ou les exigences des données.
  1. Gouvernance des Données : La gouvernance des données est un élément clé. Les politiques, les normes et les processus sont établis pour garantir la qualité, la conformité et la confidentialité des données.
  1. Collaboration avec DevOps : DataOps s'aligne souvent avec les pratiques DevOps pour assurer une intégration fluide entre le développement logiciel et la gestion des données.

En résumé, DataOps cherche à appliquer les principes agiles et les meilleures pratiques opérationnelles au cycle de vie des données, afin de créer un flux de travail plus efficace, collaboratif et automatisé.

GitOps

GitOps est une approche qui étend les principes de gestion de versions de Git aux opérations informatiques (Ops). L'idée centrale est de déclarer l'état souhaité du système à l'aide de fichiers de configuration, de les stocker dans un référentiel Git, puis d'utiliser des outils pour automatiser le déploiement et la gestion du système en fonction de cet état déclaré.

Voici quelques points clés concernant le GitOps :

  1. Déclaration de l'Infrastructure en tant que Code (IaC) : GitOps encourage la définition de l'infrastructure, des configurations et des politiques comme du code stocké dans un référentiel Git. Cela inclut souvent des fichiers YAML décrivant l'état désiré du système.
  1. Contrôle des Versions : Tous les changements de configuration et d'infrastructure sont gérés via Git. Cela offre un historique complet des modifications, des versions et des commentaires, facilitant le suivi et le retour en arrière en cas de besoin.
  1. Automatisation du Flux de Travail : GitOps repose sur l'automatisation du flux de travail, où les modifications apportées au référentiel Git déclenchent automatiquement des actions pour déployer, mettre à jour ou ajuster l'infrastructure en conséquence.
  1. Déploiement Continu (CD) : GitOps favorise le déploiement continu en automatisant le processus de déploiement dès qu'il y a une modification dans le référentiel Git. Cela conduit à des déploiements fréquents et fiables.
  1. Reconciliation Automatique : Un composant clé du GitOps est le processus de réconciliation automatique, où un opérateur ou un contrôleur surveille en permanence l'état déclaré dans le référentiel Git et ajuste automatiquement l'état réel pour correspondre à celui déclaré.
  1. Sécurité : Les approbations, les revues de code et les contrôles d'accès de Git sont appliqués pour maintenir la sécurité. Seules les modifications approuvées peuvent être fusionnées dans le référentiel principal.
  1. Observabilité : GitOps encourage également l'instrumentation et la collecte de métriques pour observer l'état du système en temps réel et diagnostiquer les problèmes.
  1. Multi-Environnement : GitOps facilite la gestion de plusieurs environnements (développement, test, production) en utilisant des branches ou des dossiers spécifiques dans le référentiel Git.
  1. Portabilité : Puisque l'infrastructure est définie comme du code, elle est portable. Les mêmes configurations peuvent être utilisées pour déployer l'application sur différentes plateformes.
  1. Résilience : En cas de défaillance, il est possible de restaurer l'état précédent en revenant à une version antérieure du référentiel Git.

En résumé, GitOps simplifie et automatise les opérations en s'appuyant sur Git comme source unique de vérité pour la configuration et l'état de l'infrastructure. Cela permet une gestion efficace, reproductible et sécurisée des systèmes informatiques.

Platform Engineering

Le Platform Engineering, ou Ingénierie de Plateforme, est une discipline qui se concentre sur la création, la gestion et l'amélioration de plateformes technologiques sur lesquelles les applications et les services peuvent être développés, déployés et exécutés de manière efficace et fiable. Cela implique souvent la conception et la maintenance d'infrastructures et d'environnements qui facilitent le développement logiciel, le déploiement continu, l'automatisation des opérations, et d'autres aspects liés à la livraison de logiciels.

Voici quelques points clés concernant le Platform Engineering :

  1. Création de Plateformes : Les Platform Engineers travaillent à la création de plateformes technologiques qui fournissent un ensemble d'outils, de services et d'infrastructures pour les équipes de développement.
  1. Automatisation : L'automatisation est un aspect central de l'ingénierie de plateforme. Cela inclut l'automatisation des processus de développement, de test, de déploiement et d'opérations pour garantir une livraison rapide et fiable des logiciels.
  1. Infrastructure en tant que Code (IaC) : L'ingénierie de plateforme s'appuie souvent sur le concept d'Infrastructure en tant que Code, où l'infrastructure est définie et gérée via du code, facilitant la reproductibilité, la gestion des versions et la scalabilité.
  1. Service aux Développeurs : Les plateformes créées par les Platform Engineers sont conçues pour fournir des services et des outils facilitant le travail des développeurs. Cela peut inclure des pipelines CI/CD, des environnements de développement, des outils de surveillance, etc.
  1. Fiabilité Opérationnelle : Assurer la fiabilité opérationnelle des plateformes est essentiel. Cela signifie garantir la disponibilité, la performance et la sécurité des services fournis par la plateforme.
  1. Sécurité : L'ingénierie de plateforme intègre également des pratiques de sécurité dès le départ. Cela inclut la gestion des identités et des accès, la protection contre les vulnérabilités, et d'autres mesures de sécurité.
  1. Évolutivité et Résilience : Les plateformes doivent être conçues pour être évolutives et résilientes, capable de gérer des charges variables et de récupérer rapidement en cas de défaillance.
  1. Adoption de Technologies Émergentes : Les Platform Engineers restent informés sur les technologies émergentes et évaluent comment les intégrer pour améliorer la productivité et la performance.
  1. Collaboration avec les Équipes de Développement : Une collaboration étroite avec les équipes de développement est cruciale. L'ingénierie de plateforme vise à comprendre les besoins des développeurs et à fournir des solutions qui améliorent leur flux de travail.
  1. Mesure et Analyse : La mesure des performances, l'analyse des données d'utilisation et le retour d'information continu sont utilisés pour améliorer constamment la plateforme.

En résumé, l'ingénierie de plateforme joue un rôle clé dans la création d'environnements technologiques favorables au développement logiciel, à la livraison continue, et à la fiabilité opérationnelle. Elle combine des compétences techniques avec une compréhension approfondie des besoins des développeurs et des opérations.

Produit et Agilité

Product Owner

Le Product Owner est un rôle clé dans la méthodologie Agile, en particulier dans le cadre de Scrum. Il représente les besoins des parties prenantes et s'assure que l'équipe de développement construit ce qui apportera la valeur maximale au produit. Le PO élabore et priorise le backlog produit, une liste ordonnée des fonctionnalités, des tâches et des améliorations à apporter au produit. Il collabore étroitement avec l'équipe de développement, clarifie les exigences, répond aux questions, et prend des décisions pour maximiser la valeur produite par l'équipe.

Product Manager

Le Product Manager est responsable de la vision globale du produit et de sa réussite sur le marché. Ce rôle est plus orienté vers la stratégie commerciale, la compréhension du marché, la gestion du cycle de vie du produit, et la collaboration avec diverses parties prenantes, y compris le marketing, la vente et la direction. Le PM définit la vision du produit, aligne les fonctionnalités avec les objectifs commerciaux, et prend des décisions éclairées sur la direction future du produit.

Owner vs Manager

La principale différence réside dans la portée de responsabilités. Le Product Owner se concentre sur la réalisation des fonctionnalités individuelles, travaillant en étroite collaboration avec l'équipe de développement, tandis que le Product Manager a une vue d'ensemble, prenant des décisions stratégiques pour le produit dans son ensemble.

Cependant, dans certaines organisations, surtout dans les startups ou les petites équipes, une seule personne peut endosser les deux rôles. Dans ce cas, cette personne doit être capable de jongler entre la gestion stratégique du produit, la vision à long terme (PM), et les détails opérationnels, la collaboration avec l'équipe de développement (PO). Cela nécessite des compétences variées, de la flexibilité et une compréhension profonde du produit et de son marché. Bien que faisable, il est crucial de maintenir un équilibre pour éviter de s'éparpiller entre les responsabilités opérationnelles et stratégiques.

Discovery

Le "discovery" en product management se réfère à la phase initiale du cycle de vie d'un produit où l'accent est mis sur la recherche, la compréhension et l'exploration des opportunités. Cette phase est cruciale pour éclairer la prise de décision stratégique et pour s'assurer que les efforts de développement sont alignés sur les besoins réels des utilisateurs et les objectifs de l'entreprise.

Voici quelques aspects clés du "discovery" en product management :

1. Compréhension des Utilisateurs :

  • Identifying User Needs: En explorant les besoins, les désirs et les défis des utilisateurs potentiels du produit.

2. Analyse du Marché :

  • Market Research: En examinant le marché pour comprendre la concurrence, les tendances émergentes et les opportunités non satisfaites.

3. Définition de la Vision et des Objectifs :

  • Vision and Goal Setting: En définissant une vision claire pour le produit et en établissant des objectifs alignés sur cette vision.

4. Exploration des Solutions Potentielles :

  • Exploring Potential Solutions: En envisageant différentes approches et solutions possibles pour répondre aux besoins identifiés.

5. Prototypage et Tests Rapides :

  • Rapid Prototyping and Testing: En créant des prototypes rapides pour valider des idées et obtenir des commentaires rapidement.

6. Évaluation de la Viabilité Technique et Financière :

  • Technical and Financial Viability Assessment: En évaluant la faisabilité technique et la viabilité financière des solutions envisagées.

7. Collaboration Interfonctionnelle :

  • Cross-functional Collaboration: En travaillant en étroite collaboration avec des équipes interfonctionnelles telles que le marketing, le développement, et d'autres parties prenantes.

8. Priorisation Initiale :

  • Initial Prioritization: En commençant à prioriser les fonctionnalités et les initiatives en fonction de leur impact attendu.

9. Itérations Rapides :

  • Rapid Iterations: En réalisant des cycles d'itérations rapides pour ajuster et améliorer les idées en fonction des retours.

10. Évaluation du Risque :
- Risk Assessment: En identifiant et évaluant les risques potentiels associés à différentes approches.

La phase de "discovery" est souvent itérative, avec des retours continus et des ajustements à mesure que de nouvelles informations émergent. Elle crée une base solide pour le développement ultérieur du produit, aidant à éviter les erreurs coûteuses en alignant les efforts sur des objectifs bien compris et des besoins réels des utilisateurs.

Ateliers de Discovery

Dans la phase de compréhension des besoins d'un projet, plusieurs ateliers peuvent être organisés pour rassembler des informations, obtenir des retours et créer une vision partagée. Voici quelques ateliers fréquemment utilisés :

1. Atelier de Cartographie d'Expérience Utilisateur (User Journey Mapping) :

  • Les participants cartographient l'ensemble du parcours de l'utilisateur avec le produit ou le service, identifiant les points de contact et les émotions associées.

2. Atelier d'Entretiens Utilisateurs (User Interviews) :

  • Des sessions d'entretiens approfondis avec des utilisateurs cibles pour comprendre leurs besoins, défis et comportements.

3. Atelier de Personas :

  • Les participants créent des personas représentant des archétypes d'utilisateurs, en se basant sur des données démographiques, des comportements et des besoins.

4. Atelier de Définition des Objectifs (Goal Definition Workshop) :

  • Les parties prenantes définissent les objectifs du projet, en se concentrant sur ce qu'elles veulent accomplir.

5. Atelier d'Alignement Stratégique :

  • Les parties prenantes alignent leurs perspectives sur la vision, les valeurs et les objectifs stratégiques du projet.

6. Atelier de Cartographie d'Écosystème (Ecosystem Mapping) :

  • Les participants identifient et cartographient les parties prenantes, les concurrents et les influences externes.

7. Atelier de Génération d'Idées (Ideation Workshop) :

  • Les membres de l'équipe génèrent créativement des idées pour répondre aux besoins identifiés.

8. Atelier de Priorisation (Prioritization Workshop) :

  • Les fonctionnalités et les initiatives sont prioritaires en fonction de leur impact attendu et de leur alignement sur les objectifs.

9. Atelier de Prototypage Rapide :

  • Les participants créent rapidement des prototypes pour visualiser des concepts et recueillir des retours précoces.

10. Atelier de Test Utilisateur (User Testing Workshop) :
- Les participants planifient et organisent des sessions de test utilisateur pour évaluer des prototypes ou des concepts.

11. Atelier de Cartographie de Flux de Travail (Workflow Mapping) :
- Les participants analysent et cartographient les flux de travail existants pour identifier des opportunités d'amélioration.

12. Atelier de Génération de Scénarios (Scenario Generation Workshop) :
- Les participants développent des scénarios d'utilisation basés sur des cas réels pour informer la conception.

13. Atelier de Cartographie des Risques (Risk Mapping Workshop) :
- Les parties prenantes identifient et évaluent les risques potentiels associés au projet.

Ces ateliers favorisent la collaboration, la créativité et la compréhension mutuelle, aidant à construire une base solide pour la suite du processus de développement. Il est souvent bénéfique d'adapter ces ateliers en fonction des besoins spécifiques du projet et des parties prenantes impliquées.

Delivery

Dans le contexte du rôle du Product Owner, le terme "delivery" peut être associé à la gestion du backlog, à la priorisation des fonctionnalités et à la livraison continue de la valeur au fil du temps.

1. Estimation du Backlog :

  • Le Product Owner est souvent impliqué dans l'estimation du backlog, c'est-à-dire évaluer la complexité et l'effort nécessaire pour mettre en œuvre chaque élément du backlog.

2. Priorisation des Fonctionnalités :

  • Le Product Owner est responsable de la priorisation du backlog en fonction de la valeur métier, des besoins des utilisateurs et des objectifs stratégiques. Cela implique de prendre des décisions éclairées sur ce qui doit être développé en premier.

3. Raffinement du Backlog :

  • Le Product Owner travaille continuellement sur le raffinement du backlog, en clarifiant les exigences, en ajoutant des détails lorsque cela est nécessaire et en s'assurant que les éléments du backlog sont prêts à être pris en charge par l'équipe de développement.

4. Interaction avec l'Équipe de Développement :

  • Le Product Owner collabore étroitement avec l'équipe de développement pour s'assurer que les user stories sont comprises, que les questions sont clarifiées et que les fonctionnalités sont développées conformément aux attentes.

5. Definition of Done (Définition de la Fin) :

  • Le Product Owner spécifie les critères de finition (Definition of Done) pour chaque user story, définissant les normes auxquelles les fonctionnalités doivent répondre avant d'être considérées comme terminées.

6. Planification des Sprints :

  • Dans un contexte agile, le Product Owner participe à la planification des sprints, aidant à déterminer quels éléments du backlog seront abordés lors de chaque itération.

7. Livraison de la Valeur Continue :

  • Le Product Owner s'efforce de livrer en continu des fonctionnalités et des améliorations qui apportent une valeur métier aux utilisateurs finaux. Cela signifie que la livraison de la valeur n'est pas uniquement associée à la fin d'un projet, mais est intégrée tout au long du processus de développement.

En résumé, dans le contexte du Product Owner, le "delivery" implique la gestion proactive du backlog, la priorisation des fonctionnalités pour maximiser la valeur métier, et une collaboration étroite avec l'équipe de développement pour garantir la réalisation réussie des objectifs du produit.

Techniques d’estimation de Backlog

Il existe plusieurs techniques que les équipes utilisent pour estimer la complexité et l'effort nécessaires pour réaliser chaque élément du backlog. Voici quelques-unes des techniques les plus couramment utilisées :

  1. Planning Poker :
    • Les membres de l'équipe reçoivent des cartes numérotées représentant des points d'effort ou de complexité (généralement de la suite de Fibonacci). Lors de l'estimation, chaque membre choisit une carte qui représente son estimation, et les cartes sont ensuite révélées simultanément. Les divergences dans les estimations sont discutées et le processus est répété jusqu'à ce qu'un consensus soit atteint.
  1. Estimation en T-Shirt Sizes :
    • Les éléments du backlog sont attribués à des tailles de t-shirt (XS, S, M, L, XL, etc.) en fonction de leur complexité ou de l'effort nécessaire. Cette approche est plus qualitative et est souvent utilisée pour des estimations rapides.
  1. Dot Voting :
    • Les membres de l'équipe reçoivent un certain nombre de points (représentés par des points, des autocollants, etc.) qu'ils peuvent distribuer entre les éléments du backlog. Cela permet une forme de vote pondéré où les membres peuvent indiquer les éléments qu'ils considèrent comme nécessitant le plus d'effort.
  1. Comparaison par Paires (Pairwise Comparison) :
    • Les éléments du backlog sont comparés deux par deux, et l'équipe décide lequel est plus complexe. Ce processus est répété jusqu'à ce que tous les éléments aient été comparés. Cela peut aider à établir une hiérarchie relative de la complexité.
  1. Estimation en Trois Points (Three-Point Estimation) :
    • Chaque élément du backlog reçoit trois estimations : la meilleure estimation (optimiste), la pire estimation (pessimiste), et la meilleure estimation probable. Ces trois points sont ensuite utilisés pour calculer une estimation probabiliste, souvent en utilisant la méthode PERT (Program Evaluation and Review Technique).
  1. Estimation Wideband Delphi :
    • Cette méthode implique une série de tours de sondages où les experts estiment individuellement chaque élément du backlog. Les estimations sont ensuite discutées entre les tours jusqu'à ce qu'un consensus soit atteint.
  1. Monte Carlo Simulation :
    • Utilise des simulations probabilistes pour estimer les plages possibles de temps ou d'effort nécessaires pour compléter un élément du backlog. Cette technique est plus avancée et peut nécessiter l'utilisation d'outils spécifiques.

Techniques de priorisation des fonctionnalités

  1. MoSCoW Method (Must have, Should have, Could have, Won't have) :
    • Les fonctionnalités sont classées dans quatre catégories en fonction de leur priorité : Must have (absolument nécessaire), Should have (important mais pas critique), Could have (souhaitable mais non essentiel), Won't have (reporté ou exclu).
  1. Analyse de la Valeur Métier (Value vs. Effort) :
    • Les fonctionnalités sont évaluées en fonction de la valeur qu'elles apportent par rapport à l'effort nécessaire pour les développer. Cela permet de se concentrer sur les fonctionnalités à fort impact avec un coût d'implémentation relativement bas.
  1. Analyse de Pareto (80/20 Rule) :
    • Applique le principe de Pareto, affirmant que 80% des résultats proviennent souvent de 20% des efforts. Les fonctionnalités qui contribuent le plus à la valeur globale sont priorisées.
  1. RICE Scoring (Reach, Impact, Confidence, Effort) :
    • Chaque fonctionnalité reçoit un score basé sur sa portée (Reach), son impact, la confiance dans l'estimation, et l'effort nécessaire. Le score total est utilisé pour déterminer la priorité.
  1. Analyse de Kano :
    • Les fonctionnalités sont classées en trois catégories : basiques (attendues mais ne procurent pas de satisfaction significative), linéaires (plus est mieux), et excitantes (surprenantes et apportant une grande satisfaction). Cela guide la priorisation en fonction de la nature des attentes des utilisateurs.
  1. Analyse de Fournisseurs Multiples (Multiple-Voting) :
    • Les membres de l'équipe ou les parties prenantes reçoivent un certain nombre de votes qu'ils peuvent attribuer aux fonctionnalités de leur choix. Cela permet une contribution collective à la priorisation.
  1. Analyse des Coûts d'Opportunité :
    • Évalue le coût d'opportunité associé à chaque fonctionnalité, c'est-à-dire ce qui pourrait être sacrifié ou reporté en priorisant une fonctionnalité plutôt qu'une autre.
  1. Analyse de Scénarios :
    • Crée des scénarios hypothétiques pour évaluer l'impact de chaque fonctionnalité sur différents aspects tels que la croissance des utilisateurs, la rétention, etc.
  1. Analyse de la Matrice de Rétention (Retention Grid) :
    • Évalue la capacité d'une fonctionnalité à retenir les utilisateurs en fonction de son importance perçue et de la facilité de sa mise en œuvre.
  1. Analyse des Coûts d'Échec (Failure Mode and Effect Analysis - FMEA) :
    • Évalue les conséquences potentielles de l'échec d'une fonctionnalité et la probabilité de cet échec, guidant la priorisation en fonction des risques.
  1. WSJF (Weighted Shortest Job First) :
  • Cette méthode attribue un score à chaque fonctionnalité en prenant en compte le Coût d'Opportunité, la Valeur Business, la Complexité, et le Temps nécessaire. Le calcul du WSJF se fait avec la formule suivante :

WSJF = Coût d’opportunité/Durée d’exécution

  • Plus le WSJF est élevé, plus la fonctionnalité est prioritaire. Cela permet de concentrer les efforts sur les fonctionnalités qui offrent le meilleur retour sur investissement en tenant compte du temps nécessaire à leur mise en œuvre. La WSJF s'inscrit particulièrement bien dans les contextes agiles à grande échelle où la priorisation doit prendre en compte à la fois la valeur business et la complexité tout en optimisant le temps de réalisation.

Scrum Master

Le Scrum Master est responsable de la compréhension, de l'adhésion et de la mise en œuvre de la méthodologie agile, en particulier le framework Scrum qu'il maîtrise parfaitement. Il veille à ce que les principes et les valeurs de la méthodologie sont respectés.

Voici une explication du métier de Scrum Master :

  1. Facilitation du Processus Scrum :
    • Le Scrum Master est responsable de faciliter le processus Scrum au sein de l'équipe de développement. Il s'assure que les principes, les règles et les pratiques Scrum sont compris et suivis.
  1. Servant Leadership :
    • Le Scrum Master adopte une approche de leadership servant, mettant l'accent sur le soutien de l'équipe pour atteindre ses objectifs. Il travaille à éliminer les obstacles, à créer un environnement propice à la collaboration et à favoriser l'autonomie de l'équipe.
  1. Suppression des Obstacles :
    • L'une des responsabilités clés du Scrum Master est de supprimer les obstacles qui pourraient entraver la progression de l'équipe. Cela peut inclure la résolution de problèmes, la coordination avec d'autres équipes ou départements, et la création d'un environnement de travail fluide.
  1. Facilitation des Cérémonies Scrum :
    • Le Scrum Master organise et facilite les différentes cérémonies Scrum, telles que les réunions de planification, les revues de sprint, les rétrospectives, etc. Il s'assure que ces événements sont efficaces et respectent les principes agiles.
  1. Communication et Collaboration :
    • Le Scrum Master favorise une communication ouverte et une collaboration efficace au sein de l'équipe et avec les parties prenantes externes. Il encourage la transparence et la confiance.
  1. Protection de l'Équipe :
    • Le Scrum Master agit comme un bouclier protecteur pour l'équipe, en s'assurant qu'elle n'est pas perturbée par des demandes externes excessives et en facilitant son focus sur la réalisation des objectifs du sprint.
  1. Amélioration Continue :
    • Il encourage l'équipe à rechercher constamment des moyens de s'améliorer. Cela peut se faire à travers des rétrospectives régulières, où l'équipe évalue son propre processus et propose des ajustements pour s'améliorer.
  1. Gestion des Conflits :
    • En cas de conflits au sein de l'équipe, le Scrum Master intervient pour les résoudre et favoriser un environnement de travail harmonieux.
  1. Éducation et Formation :
    • Le Scrum Master éduque les membres de l'équipe et les parties prenantes sur les valeurs et les principes agiles. Il peut également dispenser une formation sur Scrum si nécessaire.

Coach Agile

Le Coach Agile est expert des méthodes Agiles/Lean, son rôle va d’accompagner les équipes de devs sur la mise en place de l’agilité comme le Scrum Master, mais à l’inverse de ce dernier, il n’est pas directement rattachés aux équipes tech.

Il sait former, enseigner, être mentor sur son métier et peut intervenir en qualité de coach (réponse à un besoin spécifique.

Etant donné les compétences requises, c’est plutôt un rôle accessible après plusieurs années d’expérience (évolution d’un dev ou Scrum Master par exemple).

Les méthodes Agiles

L’agilité

L’agilité en entreprise a permis d’amener plus de souplesse dans les processus. L’utilisation d’une méthode agile permet d’adapter l’organisation du projet et de le faire évoluer en fonction des besoins. L’utilisation d’itérations courtes associée à un processus d’amélioration continue permet d’anticiper et de réagir rapidement, et éviter ainsi tout dépassement.

L’agilité consiste donc à établir une relation de confiance au sein des équipes tout en conservant une remise en question permanente des développements et des processus, de la façon à les améliorer.

Les différentes méthodes agiles

  • Scrum

Développement en mode itératif et incrémental.

Le Scrum est un cadre méthodologique de management de projet, le plus souvent utilisé pour développer des applications complexes. Le développement des produits/services s’organise en mode itératif, autour de périodes appelées “sprints”. Elle est de loin la méthode agile la plus utilisée avec 64% d’utilisateurs.

  • Kanban

Privilégie la flexibilité et le travail en flux libre.

C’est une méthode pour visualiser et améliorer en continu son processus de livraison en limitant le travail en cours. Kanban met l'accent sur la livraison continue. Elle provient de la méthode Lean et vise à améliorer le processus de travail. Dans la pratique, on matérialise sur un mur physique ou virtuel les différentes étapes du processus de réalisation d’une tâche.

Il s'agit d'un excellent outil visuel qui assure une meilleure collaboration et une meilleure communication en temps réel, ainsi qu'une bonne circulation de l'information concernant les tâches à exécuter.

La méthode Kanban reprend les principes du juste-à-temps. Chaque étape du process a une limite de tâches à effectuer qu’on ne peut pas dépasser, et s’il y a quelconque blocage sur une tâche, tout est suspendu tant que le problème n’est pas résolu.

  • Scrumban

Combine les méthodes Scrum et Kanban. Les équipes qui utilisent cette méthode profitent ainsi du cadre qu’apporte le Scrum et de la visualisation et de la flexibilité du Kanban, rendant le Scrumban très versatile.

  • Agile at Scale (SAFe)

Utilisé pour faire travailler ensemble plusieurs équipes agiles et déployer à plus grande échelle les méthodes agiles et Lean.

📊 Métiers de la Data

💡 La data (donnée) est souvent appelée "le pétrole du 21e siècle". Les métiers de la data s'occupent de récolter, nettoyer, stocker et exploiter ces données pour aider les entreprises à mieux décider.

Data Engineering

🎯 En bref : Le Data Engineer construit les "tuyaux" qui permettent aux données de circuler, d'être nettoyées et d'arriver là où elles sont utiles.

Analogie : Imaginez une ville et son réseau d'eau. Les Data Engineers sont les plombiers et ingénieurs qui construisent les canalisations, les stations de traitement, les réservoirs... pour que l'eau (= les données) arrive propre et en quantité suffisante là où elle est nécessaire — aux Data Scientists, aux tableaux de bord, aux applications.

Les Data Engineers interviennent sur l'architecture des bases de données et la préparation de la donnée. Voici les principales responsabilités de ce rôle :

  1. Ingénierie des Données (ETL/ELT) :
    • Construire des pipelines qui récupèrent des données de diverses sources, les transforment (nettoient, reformatent) et les chargent dans un entrepôt de données. Outils courants : Apache Spark, Airflow, dbt...
  1. Architecture de Données :
    • Concevoir et mettre en œuvre des architectures de données qui répondent aux besoins de l'entreprise. Cela peut inclure des choix de bases de données, de systèmes de stockage, et d'architectures distribuées, en fonction des exigences de performances, de scalabilité et de coûts.
  1. Qualité des Données :
    • S'assurer que les données sont propres, cohérentes et fiables. Des données de mauvaise qualité donnent de mauvaises décisions — c'est le principe "Garbage in, garbage out" (si on entre des déchets, on obtient des déchets en sortie).
  1. Optimisation des Performances :
    • Optimiser les performances des pipelines de données en identifiant et en résolvant les goulots d'étranglement. Cela peut impliquer l'optimisation des requêtes SQL, le partitionnement des données, la mise en cache, et d'autres techniques.
  1. Sécurité des Données :
    • Mettre en place des mécanismes de sécurité pour protéger les données tout au long de leur cycle de vie. Cela peut inclure la gestion des autorisations d'accès, le chiffrement des données, et d'autres pratiques de sécurité.
  1. Collaboration avec les Équipes :
    • Travailler en étroite collaboration avec d'autres équipes, notamment les Data Scientists, les Analystes de Données et les Ingénieurs logiciels, pour comprendre les besoins en données et fournir des solutions adaptées.
  1. Documentation et Métadonnées :
    • Documenter les pipelines de données, les schémas de données, et maintenir des métadonnées décrivant le contexte et la provenance des données. Cela facilite la compréhension et la collaboration entre les équipes.
  1. Automatisation :
    • Automatiser les processus autant que possible pour assurer la fiabilité et la reproductibilité des pipelines de données. Cela peut inclure l'utilisation d'outils d'automatisation, de planificateurs de tâches, et de solutions de surveillance.
  1. Gestion des Erreurs :
    • Mettre en place des mécanismes de gestion des erreurs pour identifier, signaler et corriger les problèmes dans les pipelines de données. Cela peut impliquer la mise en place de journaux de suivi et d'alertes en cas d'anomalies.
  1. Évolution Technologique :
    • Rester à jour avec les avancées technologiques dans le domaine du traitement de données, et évaluer comment de nouvelles technologies peuvent être appliquées pour améliorer les pipelines et les architectures existants.

Les pipelines de données

🎯 En bref : Une pipeline de données, c'est une chaîne automatisée qui collecte, transforme et déplace des données d'un point A à un point B.

Analogie parfaite : Imaginez une chaîne de production en usine. Les matières premières (données brutes) entrent d'un côté, passent par différentes machines (nettoyage, transformation, enrichissement), et un produit fini (données prêtes à l'emploi) sort de l'autre côté — de manière automatique et continue.

Une pipeline de données est un ensemble de processus et d'outils qui permettent de collecter, transformer et déplacer des données entre différents systèmes au sein d'une entreprise. Voici ses principaux composants :

  1. Ingestion : Collecte des données depuis toutes les sources — bases de données, fichiers Excel, APIs, capteurs IoT, logs d'applications...
  1. Traitement : Nettoyage (suppression des doublons, correction des erreurs), transformation (changement de format, calculs) et enrichissement (ajout d'informations complémentaires)
  1. Stockage : Conservation dans un entrepôt adapté — data warehouse pour l'analyse structurée, data lake pour le stockage brut massif
  1. Analyse et Exploration : Des outils permettent d'explorer et d'interroger les données (tableaux de bord, requêtes SQL, notebooks d'analyse...)
  1. Ordonnancement et Orchestration : Un "chef d'orchestre" (comme Apache Airflow) coordonne l'ordre et le timing des différentes étapes — comme un planning de production
  1. Monitoring et Gestion des Erreurs : Surveillance en temps réel + alertes automatiques en cas de problème dans la pipeline
  1. Sécurité : Protection des données à chaque étape, surtout si elles sont sensibles (données personnelles, financières...)
  1. Déploiement Continu : Les pipelines elles-mêmes peuvent être mises à jour régulièrement via des pratiques CI/CD pour intégrer de nouvelles sources ou corriger des bugs

Cas d'usage courants : Alimentation de tableaux de bord en temps réel, analyse de Big Data, pipelines ETL pour entrepôts de données, détection de fraudes, recommandations produits personnalisées...

Les pipelines de données contribuent à automatiser et optimiser le flux d'information au sein d'une organisation, améliorant ainsi la prise de décision et la valeur tirée des données.

🤖 Métiers de l'IA

💡 Pour bien comprendre : L'IA, c'est comme une équipe de cuisine ultra-spécialisée. Certains créent les recettes (Research Scientists), d'autres industrialisent la production (ML Engineers), d'autres servent les plats aux clients via des applications (AI Engineers), et d'autres encore gèrent la logistique de toute la cuisine (MLOps Engineers).

🔬 Data Scientist

🎯 En bref : Le Data Scientist explore les données pour en extraire des insights et construire des modèles prédictifs. C'est le pont entre la statistique, la programmation et la compréhension métier.

Analogie : C'est le détective de l'entreprise — il part de données brutes, pose des hypothèses, mène l'enquête avec des outils statistiques et livre ses conclusions sous forme de modèles ou de visualisations.

Exemple concret : Prédire le taux de churn d'une application, recommander des produits sur un e-commerce, détecter des fraudes bancaires en temps réel.

⚙️ Compétences et technologies

  • Langages : Python (pandas, NumPy, scikit-learn), R
  • Machine Learning : régression, classification, clustering, séries temporelles
  • Visualisation : Matplotlib, Seaborn, Plotly, Tableau
  • Stats : probabilités, tests A/B, inférence bayésienne
  • Data : SQL, manipulation de grands datasets, feature engineering

⚙️ ML Engineer

🎯 En bref : Le ML Engineer industrialise les modèles créés par les Data Scientists. Il les rend robustes, scalables et déployables en production — c'est le passage du prototype au produit.

Analogie : Si le Data Scientist est le chef cuisinier qui invente une recette, le ML Engineer est celui qui conçoit la chaîne de production pour fabriquer ce plat à des millions d'exemplaires sans perte de qualité.

Exemple concret : Transformer un notebook Jupyter de détection de fraude en un service API capable de traiter 10 000 transactions par seconde avec monitoring automatique.

⚙️ Compétences et technologies

  • Frameworks ML : TensorFlow, PyTorch, scikit-learn, XGBoost
  • Pipelines : MLflow, Kubeflow, Airflow
  • Serving : BentoML, TorchServe, FastAPI, ONNX
  • Infra : Docker, Kubernetes, cloud (AWS SageMaker, GCP Vertex AI)
  • Langages : Python, parfois Scala ou Java

🤖 AI Engineer

🎯 En bref : L'AI Engineer intègre des modèles d'IA (souvent des LLMs) dans des produits et applications concrètes. Il ne crée pas forcément les modèles — il sait les utiliser, les orchestrer et les connecter à des systèmes existants.

Analogie : C'est comme un électricien qui ne fabrique pas l'électricité mais sait la brancher partout où on en a besoin — APIs, chatbots, agents autonomes, recherche sémantique.

Exemple concret : Construire un assistant IA pour un service client, intégrer GPT dans une app de rédaction, créer un système RAG pour interroger une base documentaire interne.

⚙️ Compétences et technologies

  • LLMs & APIs : OpenAI API, Claude API, Mistral, Ollama
  • Orchestration : LangChain, LlamaIndex, LangGraph
  • Recherche vectorielle : Pinecone, ChromaDB, FAISS, Weaviate
  • Patterns : RAG, agents, function calling, prompt engineering
  • Langages : Python, TypeScript

🛠️ MLOps Engineer

🎯 En bref : Le MLOps Engineer assure le déploiement, la surveillance et la maintenance des systèmes ML en production. Il applique les pratiques DevOps au cycle de vie des modèles d'IA — de l'entraînement au monitoring.

Analogie : C'est le responsable maintenance d'une usine automatisée — il s'assure que les machines (modèles) tournent bien, détecte les pannes (dégradation des performances), et automatise les mises à jour.

Exemple concret : Mettre en place un pipeline CI/CD pour réentraîner automatiquement un modèle quand ses performances chutent, avec alertes et rollback automatique.

⚙️ Compétences et technologies

  • MLOps : MLflow, Weights & Biases, Kubeflow, DVC
  • Infra : Docker, Kubernetes, Terraform
  • Monitoring : Grafana, Prometheus, Evidently AI
  • Cloud : AWS SageMaker, GCP Vertex AI, Azure ML
  • CI/CD : GitHub Actions, GitLab CI appliqués aux pipelines ML

💬 LLM Engineer / Prompt Engineer

🎯 En bref : Le LLM Engineer conçoit des systèmes basés sur les grands modèles de langage — chatbots, agents autonomes, pipelines RAG. Il maîtrise l'art de guider les LLMs via des prompts et architectures adaptées pour obtenir des résultats fiables et précis.

Analogie : C'est comme un réalisateur de cinéma — le modèle (acteur) a ses capacités, mais c'est le réalisateur qui, via ses instructions (prompts, contexte, exemples), en tire la meilleure performance possible.

Exemple concret : Construire un agent capable d'interroger une base de connaissances interne, de résumer des documents juridiques ou de générer du code à partir de spécifications en langage naturel.

⚙️ Compétences et technologies

  • Prompt engineering : few-shot, chain-of-thought, structured outputs
  • Frameworks : LangChain, LlamaIndex, DSPy, Semantic Kernel
  • RAG : chunking, embedding, retrieval, reranking
  • Agents : function calling, ReAct, multi-agents (CrewAI, AutoGen)
  • Évaluation : Ragas, LangSmith, benchmarks LLM

🎓 AI Research Scientist

🎯 En bref : L'AI Research Scientist fait avancer l'état de l'art en intelligence artificielle — il conçoit de nouveaux algorithmes, architectures et méthodes, publie ses travaux et ouvre la voie aux applications futures.

Analogie : C'est le chercheur fondamental — là où les autres construisent avec les outils existants, lui invente les outils de demain. C'est le rôle le plus académique, souvent exercé dans des labos de recherche (Google DeepMind, Meta FAIR, Anthropic, Inria).

Exemple concret : Concevoir l'architecture Transformer (base de tous les LLMs modernes), proposer une nouvelle méthode d'apprentissage par renforcement, ou améliorer l'efficacité d'un modèle de diffusion.

⚙️ Compétences et technologies

  • Fondements : mathématiques avancées (algèbre linéaire, optimisation, probabilités)
  • DL : PyTorch, JAX, architectures Transformer, diffusion models
  • Publication : rédaction de papers, ArXiv, NeurIPS / ICML / ICLR
  • Expérimentation : Weights & Biases, design expérimental rigoureux
  • Domaines : NLP, vision, RL, générative AI, théorie de l'apprentissage
